# Exemplar: astral-sh/claude-code-plugins

**Score**: 97/100  |  **Date**: 2026-06-27  |  **Commit**: `f3ce88a7ba830f53afd6d944c1d0278ed318e142`

Three-skill plugin covering Astral's Python toolchain (uv, ruff, ty), notable for trigger-phrase descriptions and organizing every skill entirely around concrete use-case patterns rather than tool API documentation.

## Per-rule evidence

### R04 — Description as trigger

Each of the three skills opens with a description that pairs "Use this when" with a concrete usage context — a direct match for real user queries that invoke the skill.

> Real quote from `plugins/astral/skills/ruff/SKILL.md:3-6`:
>
> ```
> description:
>   Guide for using ruff, the extremely fast Python linter and formatter. Use this
>   when linting, formatting, or fixing Python code.
> ```

"Linting, formatting, or fixing Python code" names three distinct user activities; Claude's description-matching fires on any of them. The uv skill mirrors this: "Use this when working with Python projects, scripts, packages, or tools" — four more activation contexts in the same line. No skill wastes the description field on a generic label like "A guide to ruff."

### R05 — Body length

Rather than one monolithic "Python tools" skill, the plugin splits into three scoped files: `uv/SKILL.md` (183 lines), `ruff/SKILL.md` (135 lines), `ty/SKILL.md` (136 lines). Each stays well below the 500-line ceiling and covers exactly one tool.

> Real quote from `plugins/astral/skills/ty/SKILL.md:1-6`:
>
> ```
> ---
> name: ty
> description:
>   Guide for using ty, the extremely fast Python type checker and language
>   server. Use this when type checking Python code or setting up type checking in
>   Python projects.
> ---
> ```

Three files at 183/135/136 lines is structurally more efficient than one 454-line file: Claude loads only the skill that matches the query, not all three at once. The split earns its existence because the tools are distinct workloads — not an artificial slicing of a single topic.

### R06 — Code examples must be runnable

Every skill contains real bash commands. The migration tables show actual tool-for-tool substitution in valid shell syntax; the "Common patterns" sections use explicit `# Bad` / `# Good` pairs with real invocations.

> Real quote from `plugins/astral/skills/uv/SKILL.md:116-125`:
>
> ```
> ### pip and pip-tools → uv pip
>
> ```bash
> pip install package      → uv pip install package
> pip install -r req.txt   → uv pip install -r req.txt
> pip freeze               → uv pip freeze
> pip-compile req.in       → uv pip compile req.in
> pip-sync req.txt         → uv pip sync req.txt
> virtualenv .venv         → uv venv
> ```
> ```

Each row is a literal shell command a developer can copy. There are no placeholders, schemas, or pseudocode. The ruff skill applies the same pattern for Black, Flake8, and isort migrations, encoding 12 real command-line transitions in 30 lines.

### R08 — Patterns over theory

Each skill opens with a "When to use" section that gives a concrete, filesystem-detectable artifact signal — not a description of what the tool does, but a file whose existence tells Claude it is in a uv/ruff/ty project.

> Real quote from `plugins/astral/skills/uv/SKILL.md:12-23`:
>
> ```
> ## When to use uv
>
> **Always use uv for Python work**, especially if you see:
>
> - The `uv.lock` file
> - uv headers in `requirements*` files, e.g., "This file was autogenerated by uv"
>
> Don't use uv in projects managed by other tools:
>
> - Poetry projects (identifiable by `poetry.lock` file)
> - PDM projects (identifiable by `pdm.lock` file)
> ```

The signal is observable: `uv.lock` exists → apply uv conventions; `poetry.lock` exists → don't. The ruff and ty skills follow the same format: `[tool.ruff]` in `pyproject.toml` or a `ruff.toml` file → activate ruff; `[tool.ty]` or `ty.toml` → activate ty. Claude answers "should I use this skill here?" with a file-existence check, not a judgment call.

## Worth adopting

Pattern: **Invocation-priority ladder**. Evidence: `plugins/astral/skills/ruff/SKILL.md:30-35`:

```
## How to invoke ruff

- `uv run ruff ...` - Use when ruff is in the project's dependencies to ensure
  you use the pinned version
- `uvx ruff ...` - Use when ruff is not a project dependency, or for quick
  one-off checks
- `ruff ...` - Use if ruff is installed globally
```

The ruff and ty skills each include a short "How to invoke" section listing three invocation paths in priority order (pinned → transient → global), each with a one-line condition. No current rule captures this pattern. Why it would be a useful rule: "When a skill covers a CLI tool that ships in multiple execution contexts, include an invocation-priority list (project-pinned → ephemeral → global) so Claude doesn't default to a globally installed binary when a locked version exists in the project."
